🧭 Compass — Feature-Based Task Breakdown

4 people × 48 hours • Each person owns a vertical slice • Microsoft Track • DLW 2026
Principle: Each person owns their feature end-to-end — backend logic, agent prompts (if applicable), and the frontend component that displays it. Build independently with mock data first, integrate at Hour 30. Nobody is blocked by anyone else. Use Cursor to move fast.

⚠️ Step Zero: Agree on These Contracts (First 30 Minutes)

Before anyone writes a line of code, spend 30 minutes together defining these shared interfaces. This is what allows 4 independent streams to merge cleanly.
1. INTERACTION EVENT
The atomic unit of data. Every time a student answers a question:
student_id question_id topic subtopic difficulty student_answer correct_answer is_correct time_taken_sec timestamp
2. KNOWLEDGE MAP
Per-student state returned by the knowledge engine:
student_id → array of { topic, subtopic, mastery_score (0-1), velocity (improving/plateau/regressing), last_practiced, attempt_count, prerequisite_status }
3. AGENT OUTPUT
What the agent pipeline returns:
diagnosis (structured findings) + recommendations (actionable steps) + reasoning_trail (human-readable chain) + confidence (high/medium/low) + evaluator_verdict (approved/flagged)
4. TOPIC GRAPH
Prerequisite structure as JSON edge list:
{ "edges": [{"from": "differentiation", "to": "integration"}, ...] }
+ topic metadata: { "topic_id", "name", "subtopics": [...], "level" }
5. DEMO SUBJECT
Decide NOW and don't change. Recommended: O-Level Additional Mathematics — Ajitesh teaches this, knows the pain points, can speak to it authentically in the pitch.
FEATURE A AI Diagnosis & Maker-Checker
Owner: Ajitesh (You)
"The AI brain that diagnoses learning gaps, generates recommendations, and audits itself before speaking."
SCOPE: 3-Agent Pipeline + RAG + Reasoning Trails + Confidence Scoring
  • HR 0-2 Set up Azure OpenAI / OpenAI API. Define JSON output schemas for each agent. Write initial system prompts.
  • HR 2-6 Build Diagnosis Agent: takes a knowledge map JSON → outputs structured diagnosis (topic weaknesses, error type classification, root cause analysis via prerequisite check). Test with mock knowledge map data.
  • HR 6-12 Build Planner Agent: takes diagnosis → outputs actionable study plan with specific topics, difficulty, reasoning. Ground with RAG — embed the syllabus topic structure from Person B's graph into a vector store or context window.
  • HR 12-18 Build Evaluator Agent (maker-checker): independently reviews diagnosis + plan. Checks consistency with data, validates confidence levels, flags potential bias. Build the pipeline: Diagnosis → Planner → Evaluator → approved output OR flagged for adjustment.
  • HR 18-24 Implement error classification: for each wrong answer, LLM classifies as conceptual gap / careless mistake / time-pressure error based on question type + time taken + topic mastery context.
  • HR 24-30 Build your frontend component: a "Diagnosis & Recommendations" panel that shows the reasoning trail, confidence badges, error classifications, and evaluator verdict. This is what appears on both the student and teacher dashboards.
  • HR 30-36 Integration: connect to Person B's live knowledge engine (replace mock data). Connect your output to Person C's student dashboard and Person D's teacher dashboard.
  • HR 36-42 Polish: edge cases (new student with 0 data, student who's mastered everything, conflicting signals). Ensure reasoning trails read naturally. Tune prompts based on demo personas.
  • HR 42-48 Code freeze. Demo rehearsal. Help with pitch — explain the technical architecture to judges.

The Diagnosis Agent explains WHY (prerequisite gap in differentiation), the Planner suggests a specific action, the Evaluator confirms with "high confidence — validated against 12 data points." FEATURE B Knowledge State Engine & Topic Graph
Owner: Teammate B
"The living model of what a student knows right now — with prerequisite tracking and learning velocity."
SCOPE: Mastery Tracker + Prerequisite DAG + Learning Velocity + Data Layer + API
  • HR 0-2 Set up Python project + FastAPI. Define the interaction event schema and knowledge map response schema (from the shared contracts).
  • HR 2-6 Build the O-Level A-Math topic prerequisite graph as a NetworkX DAG. Cover ~15-20 topics with clear prerequisite edges. Store as JSON. Create topic metadata (name, subtopics, difficulty tier).
  • HR 6-12 Implement mastery tracker: per-topic score (0-1) that goes up when student answers correctly, down when wrong, weighted by question difficulty. Add temporal decay — mastery decreases towards a baseline if the student hasn't practiced in N days.
  • HR 12-18 Implement learning velocity: sliding window over last N attempts per topic. Calculate rate of change. Classify as improving / plateauing / regressing. Store velocity history for trend visualization.
  • HR 18-24 Implement prerequisite propagation: when a student is weak at Topic X, traverse the prerequisite graph to check foundational topics. Flag if root cause is upstream. Build the API endpoints: POST /interaction, GET /student/{id}/knowledge-map, GET /student/{id}/velocity, GET /topics/graph.
  • HR 24-30 Build your frontend component: the Knowledge Map Visualization. An interactive topic graph/tree where each node is colored by mastery (green ≥0.7, yellow 0.4-0.7, red <0.4). Show prerequisite edges. Show velocity indicators (↑ ↓ →) on each topic. This is the visual centerpiece of the student dashboard.
  • HR 30-36 Generate synthetic student data: create 3 demo personas with 30-50 pre-loaded interactions each, showing realistic patterns (improving student, struggling student, returning-after-break student). Integration with Ajitesh's agent pipeline.
  • HR 36-42 Polish: ensure decay works visually (demo the "student returns after 3 weeks" scenario). Edge case handling. API stability.
  • HR 42-48 Code freeze. Demo rehearsal.
✅ DEMO MOMENT
Judge sees: A beautiful topic graph where some nodes are green, some yellow, some red. Clicking a red node (e.g., "Integration by Parts") shows that its prerequisite ("Basic Integration") is also yellow — revealing the root cause. Learning velocity arrows show the student is improving in algebra but plateauing in calculus. The judge immediately understands "this isn't just a gradebook."

🚨 Risks & Mitigations

Risk: Integration at Hour 30 fails — APIs don't match. Mitigation: Agree on exact JSON schemas at Hour 0. Each person validates their output matches the contract before integration.
Risk: Azure setup takes too long. Mitigation: Have a fallback — use OpenAI API directly with your $100 credits. Azure is nicer for the Microsoft judges but the agents work the same either way.
Risk: Agent outputs are inconsistent / hallucinate. Mitigation: Use structured output (JSON mode). Pin the prompts early. The Evaluator Agent is literally your safety net — that's why you built it.
Risk: Scope creep — someone tries to add a feature at Hour 35. Mitigation: Code freeze at Hour 40 is non-negotiable. If it's not working by Hour 36, cut it, don't fix it.
Risk: Demo breaks live on stage. Mitigation: Person D pre-records a backup demo video. Have screenshots in the pitch deck as fallback. Pre-load all demo data so you're not dependent on real-time API calls.
